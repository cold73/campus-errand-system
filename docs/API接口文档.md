## 一、订单管理接口（用户端）

### 1.1 创建订单

| 项目 | 内容 |
|------|------|
| 请求方式 | POST |
| 请求路径 | `/api/order/create` |
| 功能说明 | 用户发布一笔新的跑腿订单，创建成功后返回订单 ID |

**请求体（JSON）**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| userId | Long | 是 | 发单用户 ID |
| title | String | 是 | 订单标题，非空 |
| content | String | 否 | 任务补充说明 |
| orderType | Integer | 是 | 订单类型（见枚举） |
| price | BigDecimal | 是 | 订单金额，≥0 |
| tip | BigDecimal | 否 | 跑腿小费，默认 0 |
| urgencyLevel | Integer | 否 | 紧急等级（见枚举），默认 0 |
| expectFinishTime | String | 否 | 期望完成时间，格式 `YYYY-MM-DDTHH:mm:ss` |
| receiverName | String | 是 | 收货人姓名 |
| receiverPhone | String | 是 | 收货人手机号（11 位） |
| receiverAddress | String | 是 | 收货地址 |
| pickupAddress | String | 否 | 取货地址（代拿快递场景填写） |

**返回示例**

```json
{
  "code": 200,
  "message": "success",
  "data": 10023
}
```

`data` 为新建订单的 ID（Long）。

### 1.4 取消订单

| 项目 | 内容 |
|------|------|
| 请求方式 | POST |
| 请求路径 | `/api/order/cancel` |
| 功能说明 | 用户取消订单，仅允许在订单状态为"待接单（0）"时操作 |

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|

### 2.2 接单

| 项目 | 内容 |
|------|------|
| 请求方式 | POST |
| 请求路径 | `/api/order/take` |
| 功能说明 | 跑腿员接单，订单状态由"待接单（0）"变为"已接单（1）"，同时写入跑腿员-订单关系记录 |

**请求体（JSON）**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|

**返回示例**

```json
{
  "code": 200,
  "message": "success",
  "data": 55
}
```

`data` 为跑腿员-订单关联记录的 ID（Long）。

---

### 2.3 取货确认

| 项目 | 内容 |
|------|------|
| 请求方式 | POST |
| 请求路径 | `/api/order/pickup` |
| 功能说明 | 跑腿员确认已取到货物，订单状态由"已接单（1）"变为"进行中（2）" |

**请求体（JSON）**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| orderId | Long | 是 | 订单 ID |
| runnerId | Long | 是 | 跑腿员 ID |

**返回示例**

```json
{
  "code": 200,
  "message": "success",
  "data": 10023
}
```

---

### 2.4 完成订单

| 项目 | 内容 |
|------|------|
| 请求方式 | POST |
| 请求路径 | `/api/order/finish` |
| 功能说明 | 跑腿员确认送达，订单状态由"进行中（2）"变为"已完成（3）"，同时触发虚拟收益结算，将 price+tip 写入收益明细表 |

**请求体（JSON）**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| orderId | Long | 是 | 订单 ID |
| runnerId | Long | 是 | 跑腿员 ID |

**返回示例**

```json
{
  "code": 200,
  "message": "success",
  "data": 10023
}
```

> 收益结算与订单状态更新在同一个事务内完成；通过 `uk_order_id` 唯一索引保证幂等性，重复提交不会重复结算。

## 四、AI 能力接口

### 4.1 智能定价建议

| 项目 | 内容 |
|------|------|
| 请求方式 | POST |
| 请求路径 | `/api/ai/suggest-price` |
| 功能说明 | 根据任务内容、订单类型、紧急等级，通过规则引擎给出建议价格和建议小费，并返回计价明细说明 |

**请求体（JSON）**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| title | String | 否 | 订单标题（辅助语义理解） |
| content | String | 是 | 任务描述（关键词匹配来源） |
| orderType | Integer | 是 | 订单类型（见枚举） |
| urgencyLevel | Integer | 是 | 紧急等级（见枚举） |

**返回示例**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "suggestPrice": 8.00,
    "suggestTip": 2.00,
    "reason": "基础费 3.00 + 代买商品 2.00 + 紧急附加 2.00 + 关键词[外卖] 1.00 = 8.00；建议小费 2.00"
  }
}
```

| 字段 | 说明 |
|------|------|
| suggestPrice | 建议价格（BigDecimal） |
| suggestTip | 建议小费（BigDecimal） |
| reason | 计价明细字符串，供用户参考 |

> **实现说明**：当前版本（v1）使用规则引擎：基础费 3.00 + 类型附加费 + 紧急附加费 + 关键词加价，价格区间限定在 [3.00, 50.00]。接口已预留 v2 扩展点，可无缝替换为基于大语言模型（LLM）的语义定价实现。

## 五、公共返回结构

所有接口统一返回 `Result<T>` 结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| code | Integer | 200 表示成功，其他值表示失败 |
| message | String | 提示信息 |
| data | T | 业务数据，失败时为 null |

**常见错误码**

| code | 含义 |
|------|------|
| 200 | 请求成功 |
| 400 | 参数校验失败 |
| 404 | 资源不存在 |
| 500 | 服务端处理异常 |

## 六、枚举常量说明

### 订单类型（orderType）

| 值 | 含义 |
|----|------|
| 0 | 代拿快递 |
| 1 | 代买商品 |
| 2 | 代办事务 |
| 3 | 其他 |

### 紧急等级（urgencyLevel）

| 值 | 含义 |
|----|------|
| 0 | 普通 |
| 1 | 紧急 |
| 2 | 超急 |

### 订单状态（status）

| 值 | 含义 | 可转换为 |
|----|------|----------|
| 0 | 待接单 | 1（接单）、4（用户取消） |
| 1 | 已接单 | 2（取货确认） |
| 2 | 进行中 | 3（完成） |
| 3 | 已完成 | — |
| 4 | 已取消 | — |

### 收益结算状态（income.status）

| 值 | 含义 |
|----|------|
| 0 | 未结算 |
| 1 | 已结算 |
