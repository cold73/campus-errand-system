## 一、系统概述

校园跑腿系统是一款面向在校大学生的微信小程序，提供用户发单与跑腿员接单两侧功能。用户可发布快递代取、商品代买、事务代办等类型的跑腿任务；跑腿员可在接单大厅浏览并接单，按流程完成取货、配送、完成等操作。系统设计以校园场景为核心，覆盖订单全生命周期管理，并集成了智能定价建议与订单状态可视化等辅助功能。

## 二、用户端功能

### 2.1 发布订单

用户填写订单基本信息后提交创建，支持以下字段：

- **订单标题**（必填）：简要描述任务内容
- **任务说明**（选填）：补充快递单号、特殊要求等
- **订单类型**（必填）：代拿快递 / 代买商品 / 代办事务 / 其他
- **紧急等级**（必填）：普通 / 紧急 / 超急，通过三段式按钮组选择
- **订单金额**（必填）：用户设定的跑腿费用
- **跑腿小费**（选填）：额外激励金额
- **期望完成时间**（选填）：日期 + 时间两级选择，限定不早于当前时刻
- **收货人信息**（必填）：姓名、手机号、收货地址
- **取货地址**（选填）：代拿快递场景中需填写快递存放位置

前端对所有必填字段进行本地校验，手机号格式和金额合法性校验均在提交前完成。

### 2.2 查看订单列表

用户可查看自己发布的全部订单，按发布时间倒序展示。每条记录显示：订单标题、订单编号、状态标签、紧急等级标识、金额信息。

### 2.3 查看订单详情

进入任意订单可查看完整信息，包括：

- 订单主信息（类型、金额、小费、任务说明、期望完成时间、发布时间）
- 收货信息（收货人姓名、手机号、收货地址、取货地址）
- 当前订单状态及流程进度

### 2.4 取消订单

订单处于"待接单"状态时，用户可主动取消。取消后订单状态变为"已取消"且不可恢复。系统在操作前弹出二次确认提示，防止误触。

## 三、跑腿员端功能

### 3.1 接单大厅

接单大厅展示当前所有"待接单"状态的订单，跑腿员可浏览订单内容（标题、类型、金额、紧急等级、发布时间）并选择接单。紧急和超急订单通过彩色边框突出显示，与普通订单在视觉上明显区分。

### 3.2 接单

跑腿员在接单大厅点击接单后，系统将订单状态由"待接单"变为"已接单"，同时记录接单时间，并将跑腿员与订单的关联关系写入数据库。

### 3.3 取货确认

接单后，跑腿员确认已取到货物，系统将订单状态由"已接单"变为"进行中"，并记录取货时间。

### 3.4 完成订单

送达后，跑腿员确认完成，系统将订单状态由"进行中"变为"已完成"，同时在同一事务内触发收益结算，将该订单的 `price + tip` 写入收益明细表。

### 3.5 我的接单列表

跑腿员可查看自己接到的全部任务，按接单时间倒序展示。每条任务卡片包含：订单标题、编号、状态、金额、接单时间、发布时间，以及当前可执行的操作按钮（取货 / 完成）。

## 四、个人中心功能

个人中心页面（"我的"Tab）为跑腿员提供数据概览与功能入口，主要包含：

### 4.1 用户信息卡

展示跑腿员基本信息，包括用户 ID、用户名，以及所在角色标识（RUNNER）。

### 4.2 收益总览卡

以深色大卡片形式展示跑腿员的收益数据：

- **今日收入**：通过客户端筛选当日收益明细汇总计算
- **累计收益**：调用收益汇总接口获取历史总额
- **已结算**：已完成结算的累计金额

### 4.3 接单数据统计

以三列数字形式展示：历史接单总数、进行中任务数、已完成任务数，数据来源于收益汇总接口的 `orderCount` 字段。

### 4.4 功能入口列表

提供以下快捷入口：

- 我的接单（跳转至接单任务页）
- 接单大厅（跳转至订单大厅页）
- 收益明细（展示逐笔收益记录）

## 五、订单分级功能

系统支持三级紧急程度，用于区分不同优先级的订单，便于跑腿员识别和用户表达时间需求。

### 5.1 等级定义

| 等级 | 值 | 显示文案 | 视觉风格 |
|------|----|----------|----------|
| 普通 | 0 | 普通 | 无特殊标识 |
| 紧急 | 1 | 紧急 | 橙色边框 + 橙色标签 |
| 超急 | 2 | 超急 | 红色渐变边框 + 红色标签 |

### 5.2 发布时选择

用户在发单页通过三段式按钮组选择紧急等级，选中后按钮高亮并展示对应配色，默认选中"普通"。

### 5.3 展示场景

紧急等级标识在以下页面均有展示：

- 接单大厅（hall 页）：卡片边框配色区分
- 我的订单列表（list 页）：状态胶囊旁的紧急标签
- 我的接单（tasks 页）：卡片边框配色 + 标签
- 订单详情（detail 页）：顶部标签组

## 六、虚拟收益结算功能

系统实现了一套虚拟资金流转机制，模拟跑腿员完成任务后的收益入账。

### 6.1 结算触发时机

跑腿员调用"完成订单"接口时，在数据库事务内同步写入收益记录。结算金额 = `price + tip`（订单金额 + 跑腿小费）。

### 6.2 幂等性保障

收益表（`t_runner_income`）对 `order_id` 字段建立了唯一索引（`uk_order_id`），确保同一订单不会重复结算，即使完成接口被重复调用也不会产生多条收益记录。

### 6.3 收益数据查询

提供两个接口支持收益查询：

- **收益汇总**：返回累计总收益、已结算金额、结算订单笔数
- **收益明细**：返回逐笔收益记录，包含订单 ID、金额、状态、入账时间

### 6.4 今日收入计算

今日收入在客户端通过工具函数 `sumTodayIncome()` 计算：拉取全量收益明细后，筛选 `createTime` 在当日 00:00 之后的记录求和，无需后端额外接口。

## 七、智能定价建议功能

用户在发单时，可通过"AI 估价"功能获取系统推荐的价格区间，降低定价门槛。

### 7.1 触发方式

在发单页"PRICE"字段右侧点击"✦ AI 估价"按钮，系统读取当前已填写的任务内容、订单类型、紧急等级，发起定价请求。

### 7.2 定价规则（v1 规则引擎）

当前版本采用规则引擎实现，定价逻辑如下：

| 计价项 | 金额 |
|--------|------|
| 基础费 | 3.00 元 |
| 类型附加（代拿快递） | +1.00 元 |
| 类型附加（代买商品） | +2.00 元 |
| 类型附加（代办事务） | +3.00 元 |
| 紧急附加（紧急） | +2.00 元 |
| 紧急附加（超急） | +5.00 元 |
| 关键词加价（如外卖、超市等） | +0.50～2.00 元不等 |
| 价格上下限 | [3.00, 50.00] 元 |

系统同时返回建议小费金额，以及逐项说明的计价理由字符串（如"基础费 3.00 + 代买商品 2.00 + 紧急附加 2.00 = 7.00"）。

### 7.3 应用估价

估价结果以卡片形式展示在价格字段下方，用户可点击"应用此价格"按钮，将建议价格和建议小费自动填入对应输入框，也可忽略手动填写。

### 7.4 扩展说明

接口设计预留了 v2 升级通道，后续可将 `RulePriceSuggestServiceImpl` 替换为基于大语言模型（如 DeepSeek）的实现，无需修改 Controller 层代码。

## 八、订单状态流程可视化功能

在订单详情页和跑腿员接单页，以可视化流程条展示订单当前所处的阶段，帮助用户和跑腿员直观了解订单进展。

### 8.1 详情页（4 步完整流程）

流程条位于订单标题与详情信息卡之间，展示完整的 4 个节点：

```
POSTED → TAKEN → DOING → DONE
已发布   已接单   进行中   已完成
```

各节点根据订单当前状态呈现不同样式：

| 节点状态 | 视觉样式 |
|----------|----------|
| 已过（past） | 黑色实心圆点 + 黑色实线连接 |
| 当前（active） | 橙色实心圆点 + 脉冲动画 |
| 未到（future） | 灰色空心圆点 + 灰色虚线连接 |

当订单状态为"已取消（4）"时，不显示流程条，改为展示红色警示卡片（"ORDER CANCELLED / 订单已取消"）。

### 8.2 接单任务页（3 步简化流程）

跑腿员接单任务卡片内嵌入 3 步简化流程条，位于时间信息区块与操作按钮之间：

```
TAKEN → DOING → DONE
已接单   进行中   已完成
```

样式规则与详情页一致，便于跑腿员快速判断当前任务进度。

### 8.3 实现方式

流程数据通过 `decorate()` 函数预处理生成 `steps` 数组，每个节点包含英文名、中文名、状态（`state-past` / `state-active` / `state-future`），WXML 层通过 `wx:for` 循环渲染，CSS 通过类名控制样式，视图层无逻辑计算。
